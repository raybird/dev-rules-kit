# 使用說明：new-issue 功能

## 功能概述

`new-issue` 是一個用於分析需求並建立標準化 issue 文件的技能/workflow。它會根據使用者提供的需求資訊，自動在 `docs/issues/issue-{編號}/` 目錄中建立所需的分析文件。

## 使用方式

### 方式一：自然語言對話（預設支援）
直接描述您想要建立的 issue，系統會引導您提供必要資訊。

**範例**：
```
我想要建立一個關於網站首頁載入速度優化的 issue
```

系統將會詢問：
- issue 編號
- 具體需求描述
- 相關技術細節
- 實作目標等

### 方式二：指令式介面（建議增強）
透過結構化參數直接指定 issue 資訊。

**格式**：
```
/new-issue
issue:{編號}
主題:{標題}
內容:
{詳細需求描述}
```

**範例**：
```
/new-issue
issue:123
主題:怎麼開始與調整網站
內容:
我們需要評估目前網站首頁的載入效能，並提出優化方案。
主要問題包括：
1. 首頁載入時間超過 5 秒
2. 多個未壓縮的圖片資源
3. 沒有使用快取策略
```

## 產出文件結構

執行 new-issue 後，將在 `docs/issues/issue-{編號}/` 目錄中建立以下文件：

```
docs/issues/issue-{編號}/
├── README.md              # 必備：議題概述與快速導覽
├── requirement-analysis.md # 建議：需求描述與現況分析
├── technical-analysis.md  # 建議：技術分析與實作可行性
└── implementation-plan.md # 建議：設計方案與實作步驟
```

### 文件格式規範

所有產出的文件必須遵守 `docs/AGENTS.md` 中定義的規範：

1. **日期格式**：使用 `YYYY-MM-DD`（例如：2026-05-19）
2. **日期來源**：必須使用系統當時的時間為準
3. **語言**：所有文件使用繁體中文（台灣）
4. **檔案路徑**：使用相對路徑（例如：`./requirement-analysis.md`）
5. **狀態標記**：使用清晰的狀態標記（✅ ⏳ 📝 🔄）
6. **維護紀錄**：在各文件末尾的 `## 修訂紀錄 (Changelog)` 中記錄變更

## 注意事項

1. **參數必要性**：在指令式介面中，`issue:`、`主題:` 和 `內容:` 三個欄位都是必填的
2. **編號格式**：issue 編號應該是純數字（例如：123，而非 ISSUE-123）
3. **內容格式**：`內容:` 後面的描述可以是多行文字，直到遇到下一個指令式參數或結束
4. **後續流程**：建立 issue 文件後，建議使用 `requirement-analysis`、`technical-analysis` 和 `implementation-plan` 等相應 skill 來完成完整的分析與實作規劃

## 常見問題

**Q：如果我在執行 new-issue 時忘記提供某個參數會怎樣？**
A: 系統會透過自然語言對話方式詢問您缺失的資訊。

**Q：我可以修改產出的文件嗎？**
A: 可以而且鼓勵這樣做！new-issue 只是提供一個起點，您應該根據實際情況修改和補充文件內容。

**Q：如何確保我的 issue 文件格式正確？**
A: 請參考 `docs/AGENTS.md` 中的「快速檢查清單」以及各文件類型的具體格式範例。
