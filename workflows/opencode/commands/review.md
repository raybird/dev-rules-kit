---
description: 審查當前分支提交的程式碼變更，確認是否符合需求並提出改進建議
---

## Input

| 參數 | 說明 | 範例 |
|------|------|------|
| `TASK_DESCRIPTION` | 本次變更的需求描述，可為 ticket 連結、PR title 或手動輸入 | `feat: 新增使用者登入流程` |

> 若未提供 `TASK_DESCRIPTION`，則從 commit message 或 PR title 推斷需求意圖，並在報告中標注「需求描述未提供，以下為推斷」。

---

## Review Scope

請根據以下來源取得異動內容進行審查：

- **變更來源**：`git diff dev...HEAD`
- **審查對象**：所有受影響之檔案與程式邏輯
- **目標**：確認本次修改是否符合功能需求並維持良好程式品質

---

## Pre-analysis

審查正式開始前，請先執行以下步驟：

1. 執行 `git diff dev...HEAD --stat` 取得異動檔案清單
2. **排除**自動產生或非審查目標的檔案，包含：
   - `*.lock`、`package-lock.json`、`yarn.lock`
   - `dist/`、`build/`、`.angular/`
   - 自動格式化異動（純空白或換行差異）
3. 優先審查核心邏輯檔案，再依序審查設定、樣板、測試

---

## Review Tasks

### 1. 需求與變更符合度

- 對照 `TASK_DESCRIPTION`，確認異動內容是否與需求一致
- 是否存在需求未覆蓋的修改
- 是否有超出需求範圍的過度實作

### 2. 程式正確性與邏輯

- 是否可能出現錯誤或邏輯缺陷
- 是否存在邊界情況未處理
- 是否可能破壞既有功能（regression risk）

### 3. 安全性

重點包含但不限於：

- **Firebase**：Security Rules 是否過寬、存取控制是否正確
- **Cloud Run / API**：是否有未授權存取的端點、JWT 驗證是否完整
- **Angular**：XSS / template injection、`innerHTML` 的使用是否安全
- **通用**：環境變數與 secret 是否硬編碼、資料驗證是否充分、是否有越權或敏感資料外洩風險

### 4. 程式品質

- 可讀性與命名一致性
- 架構設計與責任分離是否合理
- 效能觀察（不必要的訂閱、大量重繪、N+1 查詢等）
- 可維護性與是否符合最佳實踐

### 5. 優化與建議

- 哪些地方可以改進
- 建議替代寫法或設計方式
- 重要問題優先標記

---

## Output

請僅輸出 Markdown 審查報告，並遵循以下格式：

```markdown
# 程式碼審查報告

## 變更摘要
（列出本次異動的主要檔案與功能模組，讓閱讀者快速掌握變更範圍）

## 整體評估
（總結本次審查結果，是否建議通過、需修正或補充）

## 需求符合度
- 需求描述：{TASK_DESCRIPTION}
- 是否符合需求（是 / 部分符合 / 否）
- 缺漏或過度實作說明（若有）

## 問題與風險

> 嚴重度標籤說明：
> 🔴 MUST FIX — 邏輯錯誤、安全漏洞，必須在 merge 前修正
> 🟡 SHOULD FIX — 潛在問題、邊界未處理，建議修正
> 🟢 NICE TO HAVE — 可讀性、最佳實踐，可列入後續 backlog

- 🔴 問題 1：（說明問題所在與影響）
- 🟡 問題 2：（說明問題所在與建議）
- …

## 安全性檢查
- Firebase / Cloud Run 存取控制檢查
- Angular 前端安全性檢查
- 敏感資料與環境變數檢查
- 其他潛在安全風險

## 程式品質與效能
- 可讀性與命名
- 架構合理性
- 維護性與效能觀察

## 改進建議
- 具體建議（對應問題或獨立優化）
- 可替代作法
```
